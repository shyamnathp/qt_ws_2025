Supporting new languages in Qt?

Option 1: Rewrite Qt in New Languages

Option 1: Rewrite Qt in New Languages

  • Massive effort
  • Loss of maturity
  • Fragmentation risk

Key Insight: Rewriting Qt is not practical.

- **Massive effort**: Rewriting a framework as mature as Qt would take years of development and testing. It would require significant resources and could delay innovation. - **Loss of maturity**: Qt has decades of optimizations and stability. A rewrite would lose this maturity, and the new version might lack feature parity. - **Fragmentation risk**: Creating language-specific versions could split the community and lead to inconsistent behavior across implementations. - Conclude by emphasizing that this approach is not feasible for a framework as established as Qt.

Option 2: Create Language Bindings for Qt

Option 2: Create Language Bindings for Qt

  • Preserve Qt's maturity
  • Incremental adoption
  • Proven approach (e.g., PySide6, QtJambi etc)

Question: What made PySide6 works? The hidden hero is ...

- **Preserve Qt's maturity**: Language bindings allow us to leverage Qt's decades of stability and optimizations. - **Incremental adoption**: Developers can adopt bindings gradually without disrupting existing workflows. - **Proven approach**: PySide6 is a successful example of this strategy, enabling Python developers to use Qt effectively. - Conclude by emphasizing that this approach is practical and scalable.
